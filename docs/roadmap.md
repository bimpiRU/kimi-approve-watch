# Jarvis Hub — роадмап

Статус: Node.js-фронт работает (HTTP :8787, HTTPS :8443, Telegram-бот, OAuth/PIN).

## Фаза 0 — фундамент ✅
- Node.js сервер без зависимостей, SPA-пульт, кликабельные настройки
- OAuth GitHub/Google + упрощённый вход по PIN
- HTTPS (самоподписанный, порт 8443, make-cert.ps1)
- Telegram-бот: команды, уведомления, авто-привязка chat id
- Кнопка перезапуска, help-страница, Dockerfile

## Фаза 1 — сессии и контроль агентов (текущая)
- [ ] Панель «Сессии»: список старых сессий kimi (session_index.jsonl), поиск,
      «открыть в терминале» (wt + `kimi -S <id>`), «продолжить в фоне» (dispatch)
- [ ] Live-вывод работающего агента (автообновление лога в панели результата)
- [ ] Кнопка «Стоп» на активном запуске (kill по runId)
- [ ] Очередь задач на занятого агента (вместо отказа)

## Фаза 2 — автономность
- [ ] Автозапуск Node-сервера при входе (задача планировщика JarvisHubWeb)
- [ ] Watchdog: стабилизатор KAW пингует /api/health, поднимает при падении
- [ ] Шаблоны задач (presets): «проверь систему», «обнови репо», «почисти темп»
- [ ] Browser-уведомления о завершении агентов

## Фаза 3 — мобильность и голос
- [ ] PWA-манифест (пульт как приложение на телефоне)
- [ ] Telegram inline-кнопки: агенты и presets одним тапом
- [ ] Голосовой ввод задач (Web Speech API), мостик к jarvis-assistant
- [ ] Статистика: задачи/время/ошибки по агентам

## Фаза 4 — вынос наружу
- [ ] Домен + нормальный сертификат (Caddy/Let's Encrypt) вместо самоподписанного
- [ ] Обязательный OAuth/PIN при HOST=0.0.0.0 (запрет открытого режима наружу)
- [ ] Экспорт/импорт настроек

Полный архитектурный план: `docs/jarvis-hub-plan.md`. Анализ локальной LLM: `docs/local-llm.md`.
